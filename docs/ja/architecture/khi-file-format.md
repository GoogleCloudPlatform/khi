# KHI ファイルフォーマット v6 仕様

## 1. 動機

KHI は、バックエンドでログを処理した後、単一のファイルとして結果を出力し、このファイルをフロントエンド側で読み込むことによって可視化を行っています。このファイルベースのアプローチには以下の利点があります。

* **ログデータのポータビリティ向上**: 障害発生時の状態を十分に把握できるだけの膨大なログを保存し、後から見返すことは通常困難です。KHIファイルとして保存することで、過去の事例を容易に振り返ることができ、ポストモーテムの作成や障害の再発防止に役立ちます。
* **インタラクティブな可視化**: ログの最終的な表示と分析はフロントエンドが担います。大規模なログであっても、効率的にフォーマットされた単一ファイルを用いることで、フロントエンド側で十分な分析が可能となり、高速な動作を実現しています。

### 1.1 目指さないこと

KHIのファイルフォーマット設計において、以下の点は目標としていません。

* **無限のスケーラビリティ**: 可能な限り大規模なログを扱えるよう最適化を行いますが、DBなどのバックエンドシステムにログを保持し、レイテンシを犠牲にしてまで無限の拡張性を追求するようなクライアント・サーバーモデルの設計は採用しません。長大な期間の可視化のためのツールではなく、問題が起きた時の十分に小さな時間を分析するためのツールです。
* **v5以前の後方互換性**: 本フォーマット v6 は、初めてオープンにドキュメント化されるメジャーアップデートです。根本的なアーキテクチャの変更を伴うため、これ以前のバージョンのファイル形式との後方互換性は持ちません。

### 1.2 既存のファイルフォーマットの課題

現在のKHIファイルフォーマット（v5）には、保守性やスケーラビリティの面で以下の課題が存在していました。

* **JSONメタデータのサイズ制限**: 従来のフォーマットは、JSONによるメタデータセクションとgzippedバイナリセクションのハイブリッド構成でした。ChromeのJavaScriptランタイムにおける「500MBの文字列サイズ制限」により、大規模クラスタの検査時にはJSON部分だけでこの制限を超過し、パース不能に陥る問題がありました。
* **スキーマ管理の複雑化**: ファイルスキーマのインターフェースがフロントエンドとバックエンド間で手動で管理されており、同期や保守が困難でした。
* **スタイル情報の密結合**: 色定義やアイコンなどのフロントエンドのスタイル情報が、バックエンド側で定義された列挙型変数に依存してビルド時に生成されていました。これにより、新しいログタイプや状態を追加する際に後方互換性が損なわれやすく、プラグインによるKHIの拡張を困難にしていました。
* **フロントエンドパーサーの制約**: タイムスタンプが精度に欠けるNumber型で扱われていたほか、後方互換性を考慮した設計になっていませんでした。
* **Protobufの技術的制限**: Protobufへの移行にあたり、単一Stringフィールドやファイル全体での2GB制限、およびパーサーの64MB制限を考慮した設計が必要でした。
* **ドキュメントの不在**: ファイルフォーマットに関する公式の仕様ドキュメントが存在せず、開発や保守の障壁となっていました。

## 2. 目標

本フォーマット v6 は、上記の課題を解決するために以下の目標を達成するよう設計されています。

* **堅牢性とスケーラビリティの向上**:
  * メタデータを含むすべてのデータをProtobufへ移行する。
  * 巨大なファイルをサポートするため、データを複数の「チャンク」に分割し、ストリーミング解析に適した構造を採用する。
* **自己完結性と表現力の強化**:
  * スタイルデータ（SCSSや色の定義）をビルド時生成から、KHIファイルへの直接埋め込みへと変更し、特定バージョンのフロントエンドに依存しない表現を可能にする。
* **精度の向上とスキーマの厳格化**:
  * タイムスタンプをナノ秒単位で扱える形で処理する。
* **拡張性の確保**:
  * コンテナ構造を採用することで、既存データを破壊せずにコメントやログの追記などの新しいデータをファイル末尾へ容易に追加可能にする。

## 3. コンテナ構造

KHIファイルは単一の巨大なProtobufメッセージではなく、Protobufでビルドされた複数のチャンクデータを連結したバイナリ構造を持ちます。

| オフセット | バイト数 | コンテンツ | 説明 |
| :--- | :--- | :--- | :--- |
| `0x00` | 1 | ASCII 75 の定数 "K" | KHIファイルを識別するマジックバイト |
| `0x01` | 1 | ASCII 72 の定数 "H" | |
| `0x02` | 1 | ASCII 73 の定数 "I" | |
| `0x03` | 1 | バージョン番号 `0x06` | コンテナのファイルスキーマバージョン。内部のProtobufスキーマが変更されても、この値は不変です。 |

上記ヘッダーの後、以下のチャンクブロックが EOF まで繰り返されます。

| オフセット | バイト数 | コンテンツ | 説明 |
| :--- | :--- | :--- | :--- |
| `0x04 + offset` | 4 | チャンクサイズ | 32ビット符号なし整数の、圧縮済みチャンクバイト長 |
| `0x08 + offset` | 4 | チャンクタイプ | 32ビット符号なし整数のチャンクデータの種類 |
| `0x0C + offset` | `length` bytes | バイナリチャンクデータ | Protobufでビルドされ、gzipで圧縮されたバイナリメッセージ。 |

*1つのチャンクを読み込んだ後、次のオフセットは `現在位置 + 8 + チャンクサイズ` となります*

* **エンディアン**: すべての整数および浮動小数点数はリトルエンディアンです。

## 4. チャンク構造

各チャンクはProtobufバイナリです。チャンクのメッセージタイプは、コンテナ構造で指定された「チャンクタイプ」によって異なります。

| チャンクタイプ名 | タイプID | ルートメッセージタイプ | 定義ファイルへの参照 |
| :--- | :--- | :--- | :--- |
| File Metadata Chunk | 1 | `MetadataChunk` | [metadata.proto](../../../proto/khifile/v6/metadata.proto) |
| Interning Pool Chunk | 2 | `InterningPoolChunk` | [intern_pool.proto](../../../proto/khifile/v6/intern_pool.proto) |
| Log Chunk | 3 | `LogChunk` | [log.proto](../../../proto/khifile/v6/log.proto) |
| Timeline Style Chunk | 4 | `TimelineStyleChunk` | [style.proto](../../../proto/khifile/v6/style.proto) |
| Timeline Chunk | 5 | `TimelineChunk` | [timeline.proto](../../../proto/khifile/v6/timeline.proto) |

### 4.1 チャンク共通ルール

* **順序の非保証**: チャンクの出現順序は規定されておらず、保証されません。パーサーはファイル内の参照（文字列IDやスタイルIDなど）を正しく解決するために、全てのデータをメモリ上にロードするか、依存関係を考慮してストリーミング処理する必要があります。
* **サイズ制限とチャンク分割**: Protobufパーサーの制限を回避するため、各チャンクは圧縮前の状態で1つあたり64MBを超えないように自動的に複数に分割されます。これにより、同一タイプのチャンクが同一ファイル内に複数回出現する場合があります。
* **IDの割り当てと欠番（Orphaned ID）の許容**: 各種 ID（文字列IDやスタイルIDなど）は1から開始する数値が割り当てられ、ID値 `0` は「参照なし（NULL）」を示すために予約されています。バックエンド側で並行処理して採番する都合上、割り当てられたものの実際には使用されない「欠番（Orphaned ID）」が存在する可能性があります。このため、ID値が完全に連続している保証はないことに注意が必要です。

### 4.2 Interning Pool チャンクの最適化

[InterningPoolChunk](../../../proto/khifile/v6/intern_pool.proto) は、文字列データを重複排除して保存するチャンクです。このチャンクでは、データサイズおよび圧縮率の最適化のために以下の仕組みを採用しています。

* **文字列のソートによる圧縮効率向上**: 文字列リストを出力する際、IDの数値順ではなく文字列自体の値（辞書順）でソートして配置されます。チャンクが64MBごとに分割される際、このソートにより似た文字列が同じチャンクの近傍に配置されやすくなるため、gzip などの圧縮アルゴリズムにおける同一チャンク内での圧縮率が極めて高くなります。
* **構造化データのフラット化（InternedStruct）**: JSONやYAMLなどのネストした構造を持つマップデータ（ログのbodyなど）をシリアライズする際、キーパスをヌル文字（`\x00`）で連結してフラットなキーセット（`FieldPathSet`）に変換します。変換後、重複のないキーセット全体にユニークな ID（`FieldPathSetID`）を割り当ててプールで管理します。値はフラット化されたキーの順序に従って `Values` リストに保持されます。これにより、同一の構造を持つデータのスキーマ定義を高度に重複排除し、ファイルサイズを大幅に削減しています。
  * **具体例1（ネストされたマップとリスト）**:

    ```yaml
    map:
      key: true
    list:
      - null
      - hello
    ```

    このデータは `InternedStruct` として以下のようにシリアライズされます。
    1. **キーのフラット化**: マップのキー構造を辿り、`map` の下の `key` は `map\x00key` という1つの文字列として結合されます。ルート階層のキーは `["map\x00key", "list"]` の順序を持つセットとなり、プールで `FieldPathSetID` が割り当てられます。
    2. **値のリスト化**: `Values` リストには、上記キーセットの順序に従って値が格納されます。
       * インデックス0（`map\x00key` に対応）: `BoolValue(true)`
       * インデックス1（`list` に対応）: `ListValue` として `[NullValue(), StringValue(プール内の"hello"のID)]`
  * **具体例2（リスト内の構造体）**:
    マップがリストの要素として存在する場合、そのマップのキーは上位にフラット化されず、独立した `InternedStruct` として扱われます。

    ```yaml
    list:
      - a: 1
        b: 2
    ```

    1. ルートのキーは `["list"]` となります。
    2. `Values` のインデックス0（`list` に対応）には `ListValue` が格納されます。
    3. そのリストの最初の要素は `StructValue` となり、自身の独立したキーセット `["a", "b"]` と値 `[Int64Value(1), Int64Value(2)]` を持ちます。

### 4.3 MetadataChunk のルール

`MetadataChunk` はタイムラインの情報などに依存せず、ファイル生成時に同時に生成される付加情報です。分析対象となったログの総容量、ファイル名、適用されたログフィルターなどの情報が含まれます。この情報は将来的に拡張される可能性があるため、Proto 定義上で `oneof` フィールドを用いたメッセージ型となっています。

`MetadataChunk` が複数存在する場合、それぞれ異なる `Metadata` メッセージであることが期待されます。単一の `Metadata` メッセージは分割してシリアライズできないため、最大でも 64MB のサイズ制限があります。

### 4.4 Timeline Style Chunk のルール

`TimelineStyleChunk` には、フロントエンドがタイムラインを表示するのに必要なアイコン情報、色情報、ラベルの内容などのスタイル情報が含まれます。
このチャンクが複数存在することは想定されておらず、最大でも 64MB のサイズ制限があります。

### 4.5 その他のチャンクにおけるマージルール

`MetadataChunk` および `TimelineStyleChunk` 以外のチャンクは、データサイズが 64MB を超える場合、同一タイプのチャンクが複数に分割されて出現する可能性があります。

これらのチャンクのルートメッセージはすべて `repeated` フィールドのみで構成されています。そのため、パーサーは同一タイプのチャンクから取得した `repeated` フィールドの要素を単に連結することで、分割前の元のメッセージを復元します。
